# 全来店报价技能

本技能用于把 OpenClaw 业务表单或引导式对话输入转换为全来店标准报价文件。

## 触发场景

当用户需要以下能力时使用本技能：

- 为餐饮客户生成全来店报价单
- 根据品牌名、门店数、套餐、增值模块生成报价
- 输出对客 PDF、内部 Excel 和留档 JSON
- 让销售或售前通过业务表单一键出报价
- 当原生表单未弹出时，通过少打字的引导式对话完成报价
- 通过少打字的引导式对话完成报价

## 输入方式

本技能优先接收业务表单字段，而不是完整报价 JSON。

如果当前运行环境没有展示原生表单，则自动切换为引导式对话收集字段。

核心字段：

- 客户品牌名称
- 餐饮类型
- 门店数量
- 门店套餐
- 门店增值模块
- 总部模块
- 配送中心数量
- 生产加工中心数量
- 折扣（成交价系数）
- 人工改价原因（提供成交价系数时必填）
- 是否启用阶梯报价
- 实施服务类型
- 实施服务人天

## 工作流程

1. 执行 `python3 scripts/update_notice.py`。若有输出（说明本节点被 `check_openclaw_update.py` 升级过且当前版本尚未告知过用户），将其原样作为**本次回复的首段** Markdown 展示在用户面前，然后再进入正式报价流程。输出为空则直接跳过，不要刻意提及版本号或脚本细节。
2. 优先读取原生表单输入；若未出现表单，则进入引导式对话
3. 调用 `scripts/quote.py` 生成报价文件
4. 在当前工作目录生成三份结果文件
5. 在 OpenClaw 对话中返回”报价内容预览 + 文件下载入口”

## 交互策略

### 总原则

- 尽量让用户“点选/确认”，少量打字
- 一次只问当前缺失的关键字段，不要把所有问题堆给用户
- 对标准小连锁场景优先走默认值，减少销售录入负担
- 只在确实需要时追问数量、人天、改价原因等扩展字段

### 原生表单优先

- 若 OpenClaw 已展示原生表单，直接引导用户填写表单
- `references/openclaw_form_schema.json` 与 `references/openclaw_form_config.json` 可作为字段参考，但如果平台未自动加载，不要卡住流程

### 行业关键字 → 餐饮类型软推荐

当用户在对话中先描述了品牌或行业时，skill 应先做一次软推荐（**不自动设定**），让用户一键确认：

- 推荐为 `轻餐` 的关键词：`早餐`、`奶茶`、`茶饮`、`咖啡`、`小吃店`、`快餐`、`小碗菜`、`蛋糕店`、`卤味店`、`烘焙`、`粉面`、`饮品店`
- 推荐为 `正餐` 的关键词：`火锅`、`自助餐`、`酒楼`、`中餐厅`、`宴会厅`、`海鲜酒家`
- **需追问**（歧义，避免误判）：`茶餐厅`、`日料`、`西餐`、`小笼包`、`融合菜`、`烧烤`

对话模板：
- 命中轻餐词："听起来是 X 行业，通常按 **轻餐** 走，我先按这个推进，可以吗？"
- 命中正餐词："听起来是 X 行业，通常按 **正餐** 走，我先按这个推进，可以吗？"
- 命中歧义词："X 行业通常既可以按轻餐也可以按正餐，你们想按哪个？"
- 一句话覆盖多个关键词时，以用户主表述为准。

规则：
- **用户一键确认即可推进；不自动设定餐饮类型**，避免地区性差异造成误判（如粉面在部分地区会按正餐）。
- 若用户否认软推荐，切回"轻餐 / 正餐"的标准选择。
- 若关键词落在"需追问"桶，直接显式追问，不做猜测。

### 无原生表单时的对话式收集顺序



按以下顺序逐步收集，优先给用户选项：

1. 品牌名称
2. 餐饮类型：`轻餐` / `正餐`（若用户已描述品牌或行业，先走上节『行业关键字 → 餐饮类型软推荐』，用户确认后可跳过此问）
3. 门店数量：优先让用户直接输入数字
4. 门店套餐：若用户已指定就用指定值；未指定时**暂不锁定**，等步骤 6 收完总部模块后按下一节『默认值与少打字要求』的决策树终判（无总部模块 → `旗舰版`；含 `配送中心` / `生产加工` → `全能版` + 自动追加 `供应链基础-门店点位`）
5. 门店增值模块：根据餐饮类型给出可多选的候选项
6. 总部模块：给出可多选候选项
7. 若选择了 `配送中心`，再追问配送中心数量
8. 若选择了 `生产加工`，再追问生产加工中心数量
9. 若选择了实施服务，再追问实施服务类型与人天
10. 若用户明确要人工改价（提供了成交价系数），追问一句原因，直接用作 `人工改价原因`

### 默认值与少打字要求

- 用户只说“给我做个 10 店正餐旗舰版报价”时，应主动补全默认字段，而不是要求重新完整描述
- 若用户未指定套餐，按以下决策树判定（**适用 1–300 店全段，不分小连锁/大客户段**；`门店套餐` 是 API 必填字段，任何分段都不允许留空）：
    - **总部模块为空**：默认对应餐饮类型的 `旗舰版`（ZC-04 / QC-04），含「单门店库存」，纯单店本地运营场景。
    - **总部模块含 `配送中心` 或 `生产加工`**：默认对应餐饮类型的 `全能版`（ZC-03 / QC-03），并**自动追加** `供应链基础-门店点位` 到 `门店增值模块`。在生成前的配置摘要里显式列出这一项联动，标注"为承接总部配送/生产加工自动追加"。
    - 不要默认到 `旗舰版` + 配送中心：`单门店库存`（ZC-04 子项）与 `供应链基础-门店点位` 在 service 端是互斥业务路线，会被 400 拒。
    - 不要默认到 `供应链版`（ZC-05 / QC-05）：会丢掉「会员营销」，多数客户实际需要。若 sales 明确"客户不要会员营销"，再手动切到供应链版。
- 若用户未提门店增值模块、总部模块、实施服务，默认按“不选”处理，并在最终摘要中明确展示
- 只有出现非标准商务条件、人工改价、复杂联购、301 店及以上时，才提示转人工或要求补充说明
- 允许用户用”数字、中文名称或混合方式”回复选择结果

正餐常用候选：

- `商管接口`
- `电子发票接口`

- `宴秘书宴会版套餐`

通用候选：

- `生产加工`
- `企业微信SCRM`

轻餐常见附加：

- `商家小程序号`

正餐常见附加：

- `商家小程序号-品牌点位`

### 大客户段（31–300 店）处理规则

- 31–300 店直接调用 API，不要在客户端拦截或改写。API 会按下锚点（50 / 100 / 200 / 300）生成主报价单，并在文件里附带"夹住输入值的两档阶梯对比页"。
- 夹住规则：30 ≤ N < 50 → `[30, 50]`；50 ≤ N < 100 → `[50, 100]`；100 ≤ N < 200 → `[100, 200]`；200 ≤ N ≤ 300 → `[200, 300]`。
- 当响应里 `pricing_info.algorithm_version == "large-segment-v1"`（或 `preview.stores` 与用户输入的 `门店数量` 不一致）时，**必须**在对话回复里显式说明："你填了 N 店（落在大客户段），主报价按 M 店方案生成，随单附 M / M' 两档阶梯对比表"，**不要**让客户误以为这就是 N 店的原生报价。
- 301+ 店：API 会返 422，skill 回复 "系统当前上限 300 店，超出需走大客户人工定价通道，请联系销管"。**禁止**给出任何估算数字。
- 大客户段同样适用上节『默认值与少打字要求』的套餐默认决策树。`门店套餐` 是 API 必填字段，**任何分段都不允许留空**。

### 多档主动对比（50 / 100 / 200、100 / 200 / 300 等）

- 当用户明确要三个锚点（50 / 100 / 200 / 300 中的任意三个）一起对比时：**串行调用 `scripts/quote.py` 三次**，每次只改 `门店数量`，其余字段保持一致。
- 每次调用拿到的就是该档对应的主报价单 + 该档的两档阶梯对比页；skill 侧不做 PDF 合并，只在对话回复里用一张汇总表并排展示三档的总价与单店均价，并给出三组文件下载链接。
- 不要用 `是否启用阶梯报价=true` 替代这种多次调用 — `是否启用阶梯报价` 只在 ≤30 店段生效，返回 `[10, 20, 30]` 的一次性对比。
- `是否启用阶梯报价`：≤30 店场景下的一键对比开关，设为 true 时 API 一次返回三档；≥31 店段此字段被服务端忽略（系统始终给两档对比）。

### 对话风格要求

- 多用“我先帮你按标准场景带着走”的方式推进
- 提问尽量短，例如：“餐饮类型是轻餐还是正餐？”、“门店增值模块需要哪些？可以直接回复名称，我也可以给你候选项”
- 当信息已足够时，直接开始生成，不反复确认已明确的信息
- 生成前给一个简短配置摘要，生成后给可读预览和文件入口

## 输出结果

OpenClaw 对话输出：

- 对话中展示报价内容预览（关键字段与主要费用项）
- 对话中提供 PDF、Excel、JSON 三个文件的下载入口（或可点击文件引用）

默认文件输出：

- `品牌名-全来店-报价配置-YYYYMMDD.json`
- `品牌名-全来店-报价单-YYYYMMDD.pdf`
- `品牌名-全来店-报价单-YYYYMMDD.xlsx`

用途说明：

- PDF：对外发送客户
- Excel：内部调整与复核
- JSON：内部留档与再次生成


## OpenClaw 对话输出规范

生成报价文件后，必须在对话中同时返回以下两类结果：

1. 报价内容预览（直接可读）
- 使用清晰的小结展示关键字段，例如：品牌名称、餐饮类型、门店数量、门店套餐、折扣、总部模块、实施服务。
- 至少给出“本次配置摘要 + 主要费用项”两部分，避免只返回“已生成成功”。
- 预览段落末尾应提示：PDF/Excel 中包含所选套餐的『套餐说明』以及底部『权益类』附加条款（小程序验证次数 / 外卖接单费用），文本来源为 `references/product_catalog.md`。

2. 文件下载入口（直接可点）
- 在对话中给出 3 个文件的可下载链接或可点击文件引用：PDF、Excel、JSON。
- 链接文字应明确区分文件类型，例如：`下载报价单 PDF`、`下载报价单 Excel`、`下载报价配置 JSON`。
- 如果运行环境不支持超链接，至少返回完整文件路径，保证用户可以在对话中直接拿到文件位置。

失败时返回规则：

- 若生成失败，先用一句话说明失败原因，再给出可执行的修复建议。
- 不允许只返回堆栈信息。
- 若失败原因是原生表单未出现，不要要求用户重新安装或重新触发，直接切换为引导式对话继续完成报价。

## 约束

- 报价单抬头仅展示品牌名称
- 本期不包含硬件报价
- 若折扣超出建议范围，需要明确提示风险
- 若配置或依赖异常，优先返回清晰错误，而不是静默失败
- **禁止任何形式的折扣率外推、推演、估算**。skill 不持有因子表、不本地计算折扣；所有价格只从 API 响应 `result["preview"]` / `result["pricing_info"]` 透传。用户问"50 店大概多少"，就调 API；API 不支持就说 API 不支持，不许自己编数字。
- **对外回复不展示标价（list price）、不展示折扣率**，只展示最终价（`totals.final`）。`totals.list` 仅供内部校对、毛利分析，不进客户可见渲染。
- **门店数 ≥ 301**：直接回复"系统上限 300 店，超出请走大客户人工定价通道"，**不允许**提供任何估算数字或折扣推测。
- 若 API 返回 429 / 5xx / 网络异常：如实告知"系统暂时不可用"，**不要**回退成本地估算。
